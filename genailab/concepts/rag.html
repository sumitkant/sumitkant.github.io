What is RAG?

Grounding LLMs in Your Own Data

Large Language Models are trained on vast amounts of text — but that training has a cutoff date, and it does not include your private documents, your codebase, or your organisation's internal knowledge. When you ask an LLM a question about something it was not trained on, it either guesses (and sometimes confidently hallucinates) or says it doesn't know.

Retrieval-Augmented Generation (RAG) solves this by adding a retrieval step before generation. Instead of asking the LLM to answer from memory, you first search your own document collection for the most relevant context, then pass that context to the LLM alongside the question. The LLM's job becomes synthesis and reasoning over provided material — not recall from training.

"RAG transforms an LLM from a closed-book exam taker into an open-book researcher. The quality of the answers depends entirely on the quality of what you retrieve."

The concept was formalised in the 2020 paper "Retrieval-Augmented Generation for Knowledge-Intensive NLP Tasks" by Lewis et al. at Facebook AI Research. Since then it has become one of the most widely adopted patterns for building production AI applications — from internal Q&A bots to customer support agents to code search tools.

The RAG Pipeline

Six Stages from Document to Answer

A complete RAG system has two distinct phases: an offline indexing phase that processes your documents once, and an online query phase that runs for every user question. The indexing phase is compute-intensive but amortised across many queries; the query phase must be fast enough for real-time use.

In our implementation, embeddings are computed once per chunking strategy and shared across all three vector stores — ensuring that the comparison is purely about retrieval architecture, not embedding variance. Here is how each stage works:

1
Document Ingestion

Extract raw text from PDFs, HTML pages, markdown files, or any structured document format. The goal is a clean plain-text corpus with as little noise (page numbers, headers, footers) as possible. We use pymupdf for PDF extraction. The MUSE-Autoskill research paper yields 103,413 characters of clean text.

2
Chunking

Split the text into segments small enough for the embedding model's context window (typically 512–8,192 tokens), while preserving enough surrounding context for each chunk to be semantically meaningful on its own. This is the single most impactful decision in the pipeline — we cover three strategies in detail below.

3
Embedding

Convert each text chunk into a dense numerical vector using an embedding model. Semantically similar text produces geometrically close vectors. We use nomic-embed-text via Ollama — a local, open-weight model that produces 768-dimensional embeddings. Each chunk is truncated to 2,000 characters before embedding to avoid context overflow.

4
Indexing into a Vector Store

Persist the embeddings in a vector database optimised for similarity search. The vector store is the retrieval backbone: given a query embedding, it must quickly return the K most similar stored vectors. We benchmark ChromaDB (HNSW index), FAISS (flat exact search), and LanceDB (columnar Arrow format) — covering the three most common architectural choices.

5
Retrieval + Generation

At query time, embed the user's question using the same embedding model, retrieve the top-K most similar chunks from the vector store, and pass them as context to the LLM. The LLM's prompt becomes: "Answer this question using only the following context: [retrieved chunks]." We use Gemma 4 (gemma4:e2b-it-q4_K_M) via Ollama with top-5 retrieval.

6
Evaluation

Measure how well the system performed — did the retrieved context contain the answer? Was the generated answer faithful to that context? Did it actually address what was asked? We use two complementary evaluation methods: BERTScore (embedding-based, no LLM required) and a DeepEval-inspired LLM judge for reasoning quality.

Part 01
Chunking Strategies
What is Chunking?

The Art of Splitting Documents

An embedding model has a fixed context window — typically 512 to 8,192 tokens. You cannot embed an entire research paper in one shot. Chunking solves this by splitting the document into segments small enough to embed, while preserving enough context for each chunk to be semantically meaningful on its own.

The fundamental tension in chunking is granularity vs. coherence. Smaller chunks give you more precise retrieval but risk cutting sentences mid-thought. Larger chunks preserve more context but may retrieve too much irrelevant material around the useful sentence. The three strategies below represent three different approaches to navigating this tension.

Strategy 01 — Fixed-size Chunking

The Baseline: Character Windows with Overlap

Fixed-size chunking is exactly what it sounds like: slice the document every N characters, with an overlap window between consecutive chunks. In our implementation, we use a chunk size of 512 characters with 64 characters of overlap — so the last 64 characters of chunk i are also the first 64 characters of chunk i+1.

Why overlap matters: Without overlap, a sentence that straddles a chunk boundary gets split in half, and both halves become incomplete on their own. The overlap ensures that boundary content always appears fully in at least one chunk.
Python · chunker.py
def fixed_chunk(text: str, size: int = 512, overlap: int = 64) -> List[str]: """Split text into fixed-size character windows with overlap.""" chunks, start = [], 0 while start < len(text): end = start + size chunks.append(text[start:end].strip()) start += size - overlap # slide forward by (size - overlap) return [c for c in chunks if c]

For the MUSE-Autoskill paper (103,413 chars), fixed chunking produced 231 chunks. This is a predictable number: roughly total_chars / (chunk_size - overlap).

✓
Predictable chunk count You always know how many chunks you'll get, making memory planning straightforward.
✓
Language-agnostic Works identically on any language, domain, or document structure. No NLP tools required.
✗
Splits mid-sentence Character boundaries rarely align with sentence boundaries. Chunks may start or end mid-sentence, harming embedding quality.
✗
Ignores document structure A paragraph, table row, or section heading gets no special treatment — it's just characters in sequence.
Strategy 02 — Sentence-aware Chunking

Respecting Language Boundaries

Sentence-aware chunking uses NLTK's sent_tokenize to first split the document into individual sentences, then groups K consecutive sentences into each chunk. In our implementation, we use K=5 sentences per chunk with a 2,000-character safety cap.

The key insight is that a sentence is the minimum unit of meaning in natural language. An incomplete sentence is almost never useful for retrieval — it lacks a subject, a predicate, or both. By always respecting sentence boundaries, we ensure every chunk is at minimum a complete thought.

Python · chunker.py
def sentence_chunk(text: str, sentences_per_chunk: int = 5, max_chars: int = 2000) -> List[str]: """Group N consecutive sentences, capped at max_chars.""" sentences = nltk.sent_tokenize(text) chunks = [] for i in range(0, len(sentences), sentences_per_chunk): chunk = " ".join(sentences[i : i + sentences_per_chunk]) if chunk.strip(): chunks.append(chunk.strip()[:max_chars]) return chunks

For the same paper, sentence chunking produced 125 chunks — almost half as many as fixed chunking. This is because sentences in academic papers tend to be long, and 5 sentences often reaches the 2,000-character cap before we need to truncate.

Academic paper peculiarity: Research papers have unusually long sentences, dense with clauses and citations. A single MUSE-Autoskill paper sentence like "We found that Hermes and MUSE-Autoskill end within ~2pp of each other when using the same generated skills (58.40% vs 60.35%)" is already 130 characters.
🔍
Why fewer chunks can be better Fewer, larger chunks reduce retrieval candidates but each chunk carries more context. For cross-section questions that span multiple sentences, sentence chunking often retrieves a chunk that already contains the full answer — while fixed chunking might retrieve two partial chunks that each contain only half of it.
Strategy 03 — Semantic Chunking

Splitting on Meaning, Not Structure

Semantic chunking is the most sophisticated of the three approaches. Instead of splitting by character count or sentence count, it splits wherever the topic changes — detected by measuring cosine similarity between adjacent sentence embeddings. When similarity drops below a threshold (0.75 in our setup), a new chunk begins.

The intuition: if sentence i and sentence i+1 are semantically similar (discussing the same concept), they belong in the same chunk. If similarity drops sharply, a topic shift has occurred and a split is warranted.

Python · chunker.py
def semantic_chunk(text: str, threshold: float = 0.75, max_chars: int = 1024) -> List[str]: sentences = nltk.sent_tokenize(text) model = SentenceTransformer("all-MiniLM-L6-v2") embeddings = model.encode(sentences, show_progress_bar=False) chunks, current = [], [sentences[0]] for i in range(1, len(sentences)): sim = cosine_similarity([embeddings[i-1]], [embeddings[i]])[0][0] current_len = sum(len(s) for s in current) if sim >= threshold and current_len < max_chars: current.append(sentences[i]) # same topic → keep going else: chunks.append(" ".join(current)) current = [sentences[i]] # new topic → start new chunk if current: chunks.append(" ".join(current)) return chunks

Semantic chunking produced 617 chunks for the same paper — more than 2.5× the fixed-size count. This is the key surprise: because academic papers constantly shift between sub-topics (methods, results, related work, ablations), the similarity threshold fires frequently, creating many short, topically-focused chunks.

✓ Advantage
Each chunk is topically coherent — when you retrieve it, almost everything in the chunk is about the same concept. High precision retrieval.
✗ Drawback
More chunks = more vectors to search, more embedding time, and more risk that a cross-topic question retrieves fragmented context from many tiny chunks.
⚠️
Model dependency Semantic chunking uses all-MiniLM-L6-v2 for sentence similarity — a different model than the one used for RAG retrieval (nomic-embed-text). The quality of semantic splits depends on the quality of this secondary model. A mismatch in semantic spaces can produce splits that look wrong to the retrieval model.
Part 02
Vector Stores
How Vector Search Works

From Embeddings to Nearest Neighbours

A vector store is a database optimised for one specific query type: "given this query vector, find the K most similar stored vectors." Under the hood, all three stores we tested compute cosine similarity — the cosine of the angle between two vectors in high-dimensional space. Two vectors pointing in the same direction have cosine similarity of 1.0; perpendicular vectors score 0.0.

The difference between stores is not what similarity they compute but how they compute it — the data structure, index type, and storage backend each make different trade-offs between speed, memory, persistence, and scalability.

Why normalise before inner product? FAISS uses inner product (dot product) natively. For cosine similarity, you normalise each vector to unit length first — then inner product equals cosine similarity. All three stores in our benchmark compute equivalent cosine similarity scores.
Store 01 — ChromaDB

The Swiss Army Knife

🟠 ChromaDB
Persistent · Metadata-rich · Developer-friendly
HNSW Index
ChromaDB is an open-source embedding database designed specifically for LLM applications. It stores embeddings, documents, and metadata together in a single persistent store, backed by DuckDB and Parquet. Its default index is HNSW (Hierarchical Navigable Small World), a graph-based approximate nearest neighbour algorithm that trades a small amount of recall for dramatically faster search than brute-force.
Index Type HNSW
Persistence Disk (Parquet)
Metadata support Full
Avg latency (our run) 2.4–2.7 ms

The HNSW graph builds a layered navigable structure that lets the search algorithm "zoom in" from coarse to fine — starting in a sparse upper layer and descending to the dense bottom layer where the nearest neighbours live. This makes it sub-linear in practice: searching a million vectors takes milliseconds, not seconds.

Python · chroma_store.py
# Create collection with cosine distance metric self.collection = self.client.create_collection( name=collection, metadata={"hnsw:space": "cosine"}, ) # Add documents with pre-computed embeddings self.collection.add( ids=ids, documents=texts, embeddings=embeddings.tolist(), metadatas=[{"idx": i} for i in range(len(texts))], ) # Query: returns cosine distance (1 - similarity) results = self.collection.query( query_embeddings=[query_embedding.tolist()], n_results=top_k, ) # Convert distance → similarity: score = 1 - distance
🔧
ChromaDB gotcha we hit ChromaDB ≥0.6 rejects empty metadata dicts {} with a validation error. The fix: always pass at least one metadata key per document — we used {"idx": i}. The HNSW index also rebuilds on each insert batch, so bulk-adding all chunks at once is significantly faster than adding one by one.
Store 02 — FAISS

The Speed Demon

🟢 FAISS
In-memory · Fastest retrieval · Facebook AI Research
Flat Index (Exact)
FAISS (Facebook AI Similarity Search) is a pure C++ library exposed via Python, optimised for maximum retrieval throughput. Unlike ChromaDB and LanceDB, FAISS is not a database — it is purely a vector index. It stores no text, no metadata, and nothing on disk by default. We use IndexFlatIP (flat inner product index), which performs exact brute-force search — no approximation, no graph, just computing the dot product between the query and every stored vector.
Index Type Flat (Exact)
Persistence In-memory only
Text storage Separate list
Avg latency (our run) 0.1–0.2 ms

The reason FAISS is 10–30× faster than ChromaDB and LanceDB in our benchmark is not that it uses a smarter index — it's that our dataset is small enough (125–617 vectors) that brute-force exact search over that many vectors takes almost no time. At this scale, the overhead of graph traversal in HNSW actually costs more than a simple matrix multiply.

Python · faiss_store.py
# Normalise vectors to unit length → inner product = cosine similarity norms = np.linalg.norm(embeddings, axis=1, keepdims=True) normed = embeddings / np.where(norms == 0, 1, norms) # IndexFlatIP = exact brute-force inner product self.index = faiss.IndexFlatIP(dim) self.index.add(normed.astype(np.float32)) self.texts.extend(texts) # text stored separately in Python list # Search: returns (scores, indices) — scores are cosine similarities scores, indices = self.index.search(q, k)
💡
When FAISS wins and loses For fewer than ~50,000 vectors, FAISS flat index is the fastest option because brute-force matrix multiply is highly cache-friendly and BLAS-optimised. For millions of vectors, HNSW (ChromaDB) or IVF quantisation (FAISS with a different index type) becomes necessary — otherwise search time grows linearly with dataset size.
Store 03 — LanceDB

The Analytics Powerhouse

🔵 LanceDB
Arrow-based · Analytics-ready · Multi-modal
Lance Format
LanceDB is a newer vector store built on the Lance columnar format — Apache Arrow compatible, which means vectors, text, and metadata are all stored in the same columnar table, enabling SQL-style filtering alongside vector search. It's designed for multi-modal AI applications where you might want to filter by metadata ("retrieve only chunks from section 5") before running vector similarity.
Index Type IVF-PQ (approx)
Storage format Lance (columnar)
SQL filtering Native
Avg latency (our run) 4.4–5.9 ms

LanceDB was the slowest in our benchmark — but this is not a fair comparison at our scale. The Lance format performs best with larger datasets and benefits significantly from its built-in IVF-PQ (Inverted File with Product Quantisation) index, which we did not explicitly enable here (using scan mode for small datasets instead). LanceDB's real advantage is in production-scale hybrid search and analytics workloads where you need both vector similarity and attribute filtering.

Part 03
Evaluation Strategy
The Evaluation Challenge

Measuring Quality Without Ground Truth

Evaluating RAG quality is fundamentally harder than evaluating classification or translation. You don't have a ground-truth answer to compare against. You have a retrieved context, a generated answer, and a question — and you want to know whether the answer is faithful to the context, relevant to the question, and whether the retrieved context was even useful.

The industry-standard library RAGAS provides these metrics — but it has deep LangChain dependencies that conflict with newer library versions, and its internal LLM judge requires models to output strict JSON, which small quantised models consistently fail to do. We built our own two-layer evaluation stack instead.

✓ Layer 1: BERTScore
Embedding-based, no LLM required. Uses BERT to compute semantic similarity between the generated answer and the retrieved context / question. Fast, deterministic, and works with any language.
× Why not RAGAS?
Dependency hell + JSON parsing failures. RAGAS imports a deprecated VertexAI module from langchain-community. Its metrics require LLM outputs to be strict JSON — small models fail this consistently, silently returning 0.
🎯
Why two evaluation methods? BERTScore and our LLM judge measure different things. BERTScore catches semantic overlap but can miss logical failures. The LLM judge evaluates reasoning quality but can be too generous. Running both lets us see where they agree (robust signal) and where they diverge (interesting ambiguity) — as we'll see in the results.
Evaluation Method 01 — BERTScore

Semantic Similarity Without References

BERTScore was originally designed for machine translation evaluation — comparing a generated translation against a reference translation using contextualised BERT embeddings rather than exact word overlap (like BLEU). We repurpose it here to evaluate RAG quality without needing reference answers.

The core idea: embed both the candidate text and the reference text using DistilBERT, then compute token-level cosine similarities between every token in the candidate and every token in the reference. Precision, recall, and F1 are computed over these pairwise similarities.

🎯
Faithfulness
Does the generated answer stay grounded in the retrieved context? We compute BERTScore F1 between the answer and the concatenated context chunks.
BERTScore(answer, context)
🔗
Answer Relevancy
Does the answer address what was actually asked? We compute BERTScore F1 between the answer and the question — a proxy for topical alignment.
BERTScore(answer, question)
📐
Context Precision
Are the retrieved chunks relevant to the question? We score each chunk against the question and take the mean — measuring whether retrieval was on-target.
mean(BERTScore(chunk, question))
Important caveat: These are repurposed uses of BERTScore, not its original intent. BERTScore(answer, context) as a faithfulness proxy assumes that a faithful answer uses similar vocabulary to the source context — which is usually true but can fail when the LLM paraphrases heavily or when the context is dense with technical jargon.

We use distilbert-base-uncased as the backbone — a lighter model than the full BERT-large that BERTScore typically recommends, but appropriate for our latency requirements. The model is cached after the first load and reused across all 9 configuration evaluations.

Evaluation Method 02 — DeepEval-Inspired LLM Judge

LLM-as-Judge with Scalar Scoring

DeepEval is a popular RAG evaluation framework that uses an LLM to score retrieval and generation quality using structured rubrics. Its built-in metrics (FaithfulnessMetric, AnswerRelevancyMetric) internally ask an LLM to return a structured JSON verdict — but small models like Gemma almost never produce parseable JSON reliably.

Our implementation adopts DeepEval's scoring rubrics exactly (same 0/0.5/1.0 anchors, same three metrics) but asks the LLM to return a plain scalar instead of JSON — which every model can do reliably. We score three metrics for each Q&A result.

Python · evaluator.py — DeepEval prompt
_DEEP_FAITHFULNESS_PROMPT = """You are evaluating factual faithfulness of an AI answer. Retrieved Context: {context} Question: {question} AI Answer: {answer} Does the answer contain ONLY information supported by the context? - 1.0 = every claim is backed by the context - 0.5 = some claims supported, others added outside the context - 0.0 = answer contradicts or ignores the context Reply with ONLY a decimal number 0.0-1.0. Nothing else."""

The "easy questions" problem — and how we solved it In earlier runs with auto-generated questions, Deep scores were near 1.0 for every configuration — useless for comparison. The LLM was too generous because questions were sampled from the same document, making any retrieved context look relevant. We replaced auto-generated questions with 10 hard, hand-crafted questions requiring cross-section retrieval, specific numbers, and table lookups — which finally produced meaningful score variance.

Analysis

What the Numbers Actually Tell Us

The results surface four distinct patterns that would not have been visible without running both evaluation methods simultaneously.

Finding 01
BERTScore and LLM judge disagree on semantic chunking
Semantic chunking scores the highest on BERT faithfulness (0.778–0.782) and BERT relevancy (0.839–0.847) — but the lowest on Deep relevancy (0.65–0.75) and Deep precision (0.779). The LLM judge penalises semantic chunking because its many tiny topically-focused chunks retrieve fragmented context. BERTScore misses this because token-level overlap looks fine even when logical flow is broken.
Finding 02
FAISS is 10–30× faster with zero quality cost at this scale
FAISS retrieval averaged 0.1–0.2 ms versus 2.4–2.7 ms for ChromaDB and 4.4–5.9 ms for LanceDB — and the quality scores are statistically identical. For RAG systems with fewer than ~50,000 chunks, FAISS flat index is the optimal choice: exact search over a small dataset is faster than approximate search via HNSW graph traversal.
Finding 03
Sentence chunking wins on context precision, not faithfulness
Sentence chunking achieves the only 1.000 Deep Context Precision scores — meaning the LLM judge found all retrieved sentence-chunks directly relevant to the question. But its BERT faithfulness (0.739) is lower than both fixed (0.765) and semantic (0.778). Fewer, larger chunks retrieve context that's more on-topic but require the LLM to be faithful to more material at once.
Finding 04
Vector store choice matters far less than chunking strategy
Within each chunking strategy, all three vector stores produce nearly identical quality scores — the maximum difference is 0.006 in BERT faithfulness. But across chunking strategies, Deep relevancy ranges from 0.650 to 1.000 — a 0.35 spread. Chunking is the dominant variable; vector store is an operational choice.
Bottom line: Optimise chunking first, then choose your vector store based on operational requirements (latency → FAISS, persistence + metadata → ChromaDB, production scale + hybrid search → LanceDB). The quality impact of vector store choice, at least at this scale, is negligible.
Key Takeaways

Building Your Own RAG System

1
Start with sentence chunking as your default It's structure-aware, produces coherent units of meaning, and generalises well without threshold tuning. Upgrade to semantic chunking only for homogeneous documents (blogs, reports) — not dense academic papers.
2
Use FAISS for development, ChromaDB for production FAISS is 10–30× faster at small scale and requires zero setup. Once you need persistence, metadata filtering, or a team-friendly interface, ChromaDB's DuckDB backend is the cleanest migration path.
3
Use overlap — always Even 10–15% character overlap dramatically reduces the chance of splitting a key sentence across chunk boundaries. The storage overhead is negligible. This applies to fixed and sentence chunking both.
4
Run both embedding and LLM-based evaluation BERTScore and LLM judges measure different failure modes. Using only one will miss half the picture — especially the semantic coherence issues that BERTScore cannot detect.
5
Use hard, domain-specific questions for evaluation Auto-generated questions from the same document will always score near 1.0 on LLM judges. Hand-craft questions that require cross-section retrieval, specific numbers, and table lookups to expose real variance.
6
Keep it all local during development This entire pipeline — embedding, generation, and evaluation — ran on Ollama with no API keys. Local-first RAG development is not just feasible; it's fast enough for iterative experimentation and keeps costs at zero.
